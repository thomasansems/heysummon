# API Reference

Base URL: `http://your-instance:3445` (Docker) or `http://localhost:3435` (NPX/CLI)

All endpoints require `x-api-key` authentication unless noted.

---

## Help Requests

### Submit a request
`POST /api/v1/help`

| Field | Type | Required | Description |
|-------|------|----------|-------------|
| `apiKey` | string | ✅ | Client key (`hs_live_*`) |
| `question` | string | ✅ | Question for the human expert |
| `signPublicKey` | string | ❌ | Ed25519 public key (enables E2E) |
| `encryptPublicKey` | string | ❌ | X25519 public key (enables E2E) |
| `expertName` | string | ❌ | Route to a specific expert |
| `requiresApproval` | boolean | ❌ | Expert must approve before responding |
| `messages` | string | ❌ | Encrypted conversation history |
| `messageCount` | number | ❌ | Count of messages in history |

**Response `200`:**
```json
{ "requestId": "cm...", "refCode": "HS-A1B2C3D4", "status": "pending" }
```

---

### Get request status
`GET /api/v1/help/:requestId`

**Header:** `x-api-key: hs_live_*`

**Response `200`:**
```json
{
  "requestId": "cm...",
  "refCode": "HS-A1B2C3D4",
  "status": "responded",
  "response": "No, do not delete it.",
  "respondedAt": "2026-02-28T14:00:00Z"
}
```

---

### Get request by ref code
`GET /api/v1/requests/by-ref/:refCode`

**Header:** `x-api-key`

Look up any request using its `HS-XXXXXXXX` reference code.

## Key Exchange

### Exchange public keys
`POST /api/v1/key-exchange/:requestId`

**Header:** `x-api-key: hs_exp_*`

Expert sends their public keys to enable E2E encryption with the consumer.

```json
{
  "signPublicKey": "hex-encoded-Ed25519-public-key",
  "encryptPublicKey": "hex-encoded-X25519-public-key"
}
```

**Response `200`:**
```json
{ "success": true }
```

## Messages

### Send a message
`POST /api/v1/message/:requestId`

**Header:** `x-api-key`

```json
{
  "from": "consumer",
  "plaintext": "Follow-up question here"
}
```

Or encrypted:

```json
{
  "from": "consumer",
  "ciphertext": "...",
  "iv": "...",
  "authTag": "...",
  "signature": "...",
  "messageId": "uuid"
}
```

---

### List messages
`GET /api/v1/messages/:requestId`

**Header:** `x-api-key`

Returns all messages in the conversation, ordered by creation time.

## Conversations

### Close a request
`POST /api/v1/close/:requestId`

**Header:** `x-api-key: hs_exp_*`

Closes the request. Expert key required.

---

### Rate a response
`POST /api/v1/rate/:requestId`

**Header:** `x-api-key`

```json
{ "rating": 4, "feedback": "Very helpful, thank you" }
```

**Response `200`:**
```json
{ "success": true, "rating": 4 }
```

Rating is `1`–`5`. Each request can only be rated once — a second call returns `409`.

---

### Report timeout
`POST /api/v1/help/:requestId/timeout`

**Header:** `x-api-key`

Tells the server the consumer gave up waiting. The request stays visible on the
dashboard so the expert can still respond later.

## Expert availability

Experts can configure availability windows. If a request is submitted outside
the expert's available hours, the API returns:

```json
{
  "rejected": true,
  "nextAvailableAt": "2026-03-27T09:00:00Z",
  "message": "Expert is currently unavailable. Next available at 09:00 CET."
}
```

Availability is configured per expert profile in the dashboard:
- **Quiet hours** — start/end time in the expert's timezone
- **Available days** — which days of the week the expert accepts requests

## Summoning context

Summoning context is **per-client** -- each client connection can have its own advisory
text (max 500 characters) that tells AI agents when to summon the expert. The context
is set during client setup and stored on the `SetupToken`:

1. Expert enters context in the onboarding wizard when connecting a client
2. Passed in the `POST /api/v1/setup-link` body as `summonContext`
3. Stored on `SetupToken.summonContext` for that specific client
4. Embedded as `HEYSUMMON_SUMMON_CONTEXT` in the consumer's `.env`
5. Referenced by the skill's "When to summon" section at runtime

Recently used contexts are saved to the expert's `recentSummonContexts` (JSON array,
max 10 entries, most recent first) for quick reuse when setting up new clients.

| Field | Location | Type | Max | Description |
|-------|----------|------|-----|-------------|
| `summonContext` | `POST /api/v1/setup-link` body | string \| null | 500 | Per-client advisory guidelines for AI |
| `recentSummonContexts` | `GET /api/experts/:id` response | string (JSON) \| null | 10 entries | Recently used contexts for quick reuse |

## Webhooks

HeySummon can send webhook notifications to external endpoints when events occur.

**Payload:**

```json
{
  "type": "new_request",
  "requestId": "cm...",
  "refCode": "HS-A1B2C3D4",
  "question": "Should I proceed?"
}
```

**Event types:** `new_request`, `new_message`, `responded`, `closed`, `client_timeout`, `keys_exchanged`

**Security:** Every webhook includes an `X-HeySummon-Signature` header containing
an HMAC-SHA256 signature of the payload body. Verify this against your webhook secret
before trusting the payload.

## Request lifecycle

Requests move through these states:

```
pending → reviewing → active → responded → closed
                                    ↗
pending → expired (72h TTL)
pending → cancelled
```

| State | Meaning |
|-------|---------|
| `pending` | Submitted, waiting for expert |
| `reviewing` | Expert has opened the request |
| `active` | Expert is composing a response |
| `responded` | Expert sent a response |
| `closed` | Consumer acknowledged and closed |
| `expired` | No response within 72 hours |
| `cancelled` | Explicitly cancelled |

## Error codes

| Code | Meaning |
|------|---------|
| `400` | Validation error |
| `401` | Missing or invalid API key |
| `403` | Key not authorized for this resource |
| `404` | Not found |
| `413` | Payload too large (>1MB) |
| `422` | Content blocked by safety filter (PII, XSS) |
| `429` | Rate limited |
| `500` | Internal server error |
