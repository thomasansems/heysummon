# Changelog

## Unreleased

### Fixed
- **Multi-user dashboard onboarding loop.** Users added after the first instance setup are no longer bounced into the onboarding wizard. The `/api/onboarding/status` route now returns platform-global expert and API key counts (previously scoped per-user), so `OnboardingGuard` recognises that the platform is already configured and lets additional users through to the dashboard. Their `user.onboardingComplete` flag is auto-flipped to `true` on first call so the check is cheap on subsequent page loads (HEY-226)

### Added
- **`POST /api/v1/acknowledge/:requestId` + shared ack service.** Notification-mode requests (`responseRequired: false`) now have a dedicated terminal verb. The route accepts either a dashboard session cookie or `x-api-key` (expert key or consumer key scoped to the request) and sets `status = "acknowledged"` + `acknowledgedAt`. It's idempotent ŌĆö re-acking returns the original timestamp. A new module `src/services/notifications/acknowledge.ts` (`acknowledgeNotification`) is the single code path used by the route, the dashboard ack button (HEY-371), and the upcoming Telegram/Slack/OpenClaw webhook handlers (HEY-368/369/370). `POST /api/v1/close/:requestId` now rejects notification-mode requests with `409 NOT_APPLICABLE` (stays idempotent on already-`acknowledged` or `expired`), so the two verbs are cleanly separated (HEY-365)
- **`heysummon "<question>"` one-liner.** New `ask` subcommand in the CLI (`@heysummon/app`) lets you gate any shell command on a human decision: `heysummon "Proceed?" && ./deploy.sh prod`. Positional sugar means the `ask` keyword is optional ŌĆö the first non-flag argument is treated as the question. Flags: `--timeout <duration>` (default `5m`), `--channel <name>`, `-q`/`--quiet`, `--require approve` (reserved; default behaviour). Credentials come from `HEYSUMMON_API_KEY` + `HEYSUMMON_BASE_URL` in the shell env, falling back to `~/.heysummon/.env`. The landing page and quick-start both showcase the new flow (HEY-361).

  **Exit-code contract:**

  | Exit | Meaning |
  |---|---|
  | `0` | Approved (or expert replied). Expert's plaintext reply, if any, is on stdout. |
  | `1` | Denied |
  | `2` | Timeout ŌĆö no decision within the window. `reportTimeout` is called on the server. |
  | `3` | Config / auth error (missing creds, bad flag, invalid or unauthorised key). |
  | `4` | Network / server error. |

  Under the hood, the CLI delegates request submission and polling to the consumer SDK's `submit-and-poll` with a new `--ask-mode` flag, so existing skill scripts that rely on the `APPROVED`/`DENIED` stdout contract are unaffected.

  **Trust boundary:** expert replies are untrusted input. Quote them before passing them to shells or interpreters ŌĆö `$(heysummon -q ŌĆ”)` will interpolate whatever the expert typed, including shell metacharacters.

- **n8n community node:** new `n8n-nodes-heysummon` package adds a HeySummon node to any n8n workflow. Two operations -- `Summon` (blocking ask with timeout) and `Get Status` (poll an existing request). Auth via a `HeySummon API` n8n credential type. Self-hosted base URL is a first-class input. End-to-end encryption is on by default; the lifecycle constraint between `Summon` and `Get Status` is documented on the integration page. The package ships under MIT to qualify for n8n's Verified Community Nodes program

### Changed
- **Messaging:** headline USP updated from "end-to-end encrypted" to "platform agnostic" across the landing page, marketing site, READMEs, and docs home. Encryption remains a core security feature and is still documented in `/security/` and `/consumer/encryption`, but the primary positioning now emphasizes that HeySummon works with any agent framework, any model, and any notification channel ŌĆö no SDK lock-in, no vendor tie
- **Integrations IA:** `website/pages/integrations/` reorganised into **Channels** (Twilio) and **Orchestrators** (n8n now; LangGraph, CrewAI, Make, Zapier reserved as coming-soon). Top nav label updated from "Other Integrations" to "Integrations"
- **`clients/overview.mdx`:** adds a short clarifier distinguishing AI coding clients (Claude Code, Codex, Gemini, OpenClaw) from workflow orchestrators (n8n), with a link forward to the orchestrators section
- **Consumer SDK:** `HeySummonClientOptions` now accepts optional `userAgent` and `extraHeaders` fields, threaded through every outbound request. Existing callers are unaffected. Used by the n8n node to attribute outbound traffic per PRD ┬¦4.8

## v0.2.21 -- 2026-04-09

### Security
- **Auth brute-force protection:** authentication endpoints (`/api/auth/*`) are now rate-limited to 10 requests/minute per IP. Previously these endpoints were exempt from rate limiting entirely
- **Lies-in-the-Loop mitigation:** all user-controlled content in Telegram notifications is now escaped to prevent HITL dialog forging attacks where crafted messages could manipulate expert perception
- **Health endpoint hardening:** `/api/health` no longer exposes infrastructure details (database type, rate limiting strategy, session mechanism). Returns only status and timestamp
- **IP address redaction:** API error responses no longer leak client IP addresses in hints
- **CORS tightening:** CORS headers are now only sent when the request origin matches the allowlist. Unknown origins receive no CORS headers and preflight requests return 403
- **X-Powered-By removed:** Next.js no longer sends the `X-Powered-By` header that revealed the tech stack
- **Caddy proxy hardening:** added `trusted_proxies private_ranges` to prevent X-Forwarded-For spoofing, added X-Frame-Options DENY, Permissions-Policy, and header stripping
- **Validation error redaction:** Zod validation details (field names, types) are only returned in development mode. Production returns a generic "Validation failed" message
- **E2E bypass production-gated:** the rate limit bypass header for E2E tests is now disabled in production
- **Error message hardening:** API key scope errors and help request validation errors no longer reveal internal field names or key capabilities

## v0.2.18 -- 2026-04-08

### Fixed
- **NPX installer:** generated `.env` had `DATABASE_URL="file:./prisma/heysummon.db"`, which Prisma resolves relative to `schema.prisma` and ended up creating the SQLite file at `~/.heysummon/app/prisma/prisma/heysummon.db`. Migrations wrote there, but the runtime looked at the correct location and silently 500'd. Fresh installs effectively had no working database, leaving users stuck on the login page instead of the first-time signup screen
- **NPX installer:** `ensureAppEnv` now always overwrites `~/.heysummon/app/.env` from the canonical `~/.heysummon/.env`, so config edits actually take effect (previously the copy was skipped if the destination existed)
- **Auth flags endpoint:** `GET /api/auth/flags` now catches database errors and returns `dbError: true` instead of throwing a 500. The login page renders a clear "Database not reachable" message so a misconfigured install no longer silently degrades into an inert login form

### Breaking
- **NPX installer:** existing broken installs from earlier 0.2.x are not auto-migrated. Beta users with a nested `~/.heysummon/app/prisma/prisma/heysummon.db` should either reinstall (`rm -rf ~/.heysummon && npx @heysummon/app`) or manually move the file

## Approval Buttons -- 2026-04-04

### Added
- **Approval buttons in Telegram**: Requests with `requiresApproval: true` send inline Approve/Deny buttons via Telegram `InlineKeyboardMarkup`. Callback queries update the request and confirm the decision in-chat.
- **Approval buttons in Slack**: Requests with `requiresApproval: true` send Block Kit action buttons (green Approve, red Deny). Clicking a button updates the original message to show the decision and removes the buttons.
- **OpenClaw approval webhook**: New `POST /api/adapters/openclaw/:channelId/webhook` endpoint receives `approve`, `deny`, and `reply` actions from OpenClaw. Supports HMAC-SHA256 signature verification via `X-OpenClaw-Signature` header.
- **Consumer SDK `requiresApproval` option**: `submitRequest({ requiresApproval: true })` triggers approval buttons instead of free-text responses. Decisions are delivered as message events, polled via `getPendingEvents()`.
- **Database fields**: `requiresApproval` (boolean) and `approvalDecision` (string, `"approved"` or `"denied"`) on the HelpRequest model.

### Changed

**Platform**
- Help request submission and response flow

**Documentation**
