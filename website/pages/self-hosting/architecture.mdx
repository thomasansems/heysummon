# Architecture

HeySummon is designed around security, reliable delivery, and content-blind relay. The platform routes messages without reading encrypted content, though it does see metadata (reference codes, timestamps, status) by design.

## App (Next.js)

The single service that handles everything:

- App Router API routes under `/api/*` and `/api/v1/*`
- Dashboard UI for human experts
- Auth via NextAuth.js (email/password + optional OAuth)
- Prisma ORM with PostgreSQL (or SQLite for NPX installs)
- Built-in content safety middleware

### Content safety (built-in)

Content safety runs as in-process middleware on API routes. Every incoming request is automatically scanned before processing.

**What it does:**
- Scans content for XSS and strips HTML tags
- Defangs URLs in content (`https://` becomes `hxxps://`, dots become `[.]`)
- Detects PII: credit cards, SSNs, BSNs, emails, phone numbers
- Blocks requests containing credit card numbers or SSN/BSN patterns (returns 422)

## Event delivery

Events are delivered via **HTTP polling**. Clients poll `GET /api/v1/events/pending` to receive updates.

```
Provider Watcher  -->  GET /api/v1/events/pending
Dashboard Browser -->  GET /api/v1/events/pending
```

The polling endpoint:
- Authenticates the client (API key or session cookie)
- Resolves allowed topics per client
- Returns pending events as JSON

## Request flow

```

  |
  |  POST /api/v1/help (with x-api-key)
  v
App
  |  Content safety middleware scans and sanitizes
  |  Stores encrypted request, generates ref code
  |  Creates event on provider topic
  v
Provider watcher / Dashboard (via polling)
                       |
                       |  Human reads, responds
                       v
                   App
                       |  Updates request status, creates event
                       v
                   AI Agent (via polling)
```

## E2E encryption flow

```
Consumer (AI Agent)           Platform (blind relay)         Provider (Human)

Generate Ed25519 keypair
Generate X25519 keypair
                    ---- POST /api/v1/help ---->
                         (signPubKey, encryptPubKey,
                          question)
                                                ---- notification ---->
                                                              Generate keypairs
                                                <-- POST /key-exchange --
                                                     (signPubKey, encryptPubKey)

  Both sides derive shared secret via X25519 DH — never transmitted
  Per-message keys via HKDF-SHA256, encrypted with AES-256-GCM
  Every message signed with Ed25519

                    <---- encrypted messages (via polling) --------------
Verify signature,
decrypt with derived key
```

The platform stores only ciphertext and public keys. Private keys never leave client devices, and the shared secret is computed locally — the platform cannot decrypt messages.

## Security model

| Threat | Mitigation |
|--------|-----------|
| Unauthenticated requests | Platform validates `x-api-key` with HMAC-SHA256 comparison |
| Malicious content | Built-in middleware scans for XSS, PII, URLs on every request |
| Data exposure | E2E encryption — platform stores ciphertext only |
| Brute force | Rate limiting per IP at the Platform edge (60 req/min API, 30 legacy polling, 300 events polling) |
| Key compromise | Key rotation with 24h grace period, IP binding, device tokens |
