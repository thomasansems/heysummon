# Consumer API Reference

All endpoints use `x-api-key` authentication with a client key (`hs_live_*`).

---

## Submit a help request

```
POST /api/v1/help
```

Sends a help request to a human expert. Returns immediately with a request ID.

### Body

```json
{
  "apiKey": "hs_live_abc123...",
  "question": "Should I delete the user's account permanently?",
  "signPublicKey": "base64-encoded-Ed25519-public-key",
  "encryptPublicKey": "base64-encoded-X25519-public-key",
  "messages": "encrypted-conversation-history",
  "messageCount": 5
}
```

| Field | Required | Description |
|-------|----------|-------------|
| `apiKey` | ✅ | Your client API key |
| `question` | ✅ | The question for the human expert |
| `signPublicKey` | ❌ | Ed25519 public key — enables E2E encryption |
| `encryptPublicKey` | ❌ | X25519 public key — enables E2E encryption |
| `requiresApproval` | ❌ | When `true`, the expert sees Approve/Deny buttons instead of a free-text reply field |
| `messages` | ❌ | Encrypted conversation history for context |
| `messageCount` | ❌ | Number of messages in `messages` |

### Response

```json
{
  "requestId": "cmxxx...",
  "refCode": "HS-A1B2C3D4",
  "status": "pending",
  "expiresAt": "2026-03-03T10:00:00.000Z"
}
```

## Poll request status

```
GET /api/v1/help/:requestId
```

Check the status of a help request. Use this to poll for a response.

<Callout type="info">
  For event-based updates, use the [polling endpoint](/consumer/realtime) (`GET /api/v1/events/pending`).
</Callout>

### Response (pending)

```json
{
  "requestId": "cmxxx...",
  "refCode": "HS-A1B2C3D4",
  "status": "pending",
  "expiresAt": "2026-03-03T10:00:00.000Z"
}
```

### Response (responded)

```json
{
  "requestId": "cmxxx...",
  "refCode": "HS-A1B2C3D4",
  "status": "responded",
  "response": "No, do not delete the account.",
  "respondedAt": "2026-02-28T14:30:00.000Z"
}
```

## Send a message

```
POST /api/v1/message/:requestId
```

Send a follow-up message in an ongoing conversation.

### Body

```json
{
  "from": "consumer",
  "plaintext": "Follow-up: what about archiving instead?"
}
```

| Field | Required | Description |
|-------|----------|-------------|
| `from` | ✅ | Must be `"consumer"` for client keys |
| `plaintext` | ✅* | Plaintext message |
| `ciphertext` / `iv` / `authTag` / `signature` | ✅* | Encrypted message fields |

*Either `plaintext` or all encrypted fields required.

### Response

```json
{
  "success": true,
  "messageId": "uuid...",
  "createdAt": "2026-02-28T14:35:00.000Z"
}
```

## Approval workflow

When you submit a request with `requiresApproval: true`, the expert receives Approve/Deny buttons in their notification channel (Telegram, Slack, or OpenClaw) instead of a free-text reply prompt.

### Submitting an approval request

```typescript
const result = await client.submitRequest({
  question: "Should I proceed with the $5,000 purchase?",
  requiresApproval: true,
});
```

### Polling for the decision

The approval decision arrives as a message event, just like a regular response. Poll with `getPendingEvents()` or check `getRequestStatus()`:

```typescript
// Option 1: Poll events
const events = await client.getPendingEvents();
// Events include type "new_message" when a decision is made

// Option 2: Check status directly
const status = await client.getRequestStatus(result.requestId!);
// status.status === "responded" when the expert has decided
```

### Reading the decision

Fetch the message history to get the decision:

```typescript
const messages = await client.getMessages(result.requestId!);
// The expert's message contains "approved" or "denied"
```

<Callout type="info">
  The `approvalDecision` field on the request is also set to `"approved"` or `"denied"` once the expert acts. Each request can only receive one decision — subsequent button presses are ignored.
</Callout>

## Close a request

```
POST /api/v1/close/:requestId
```

Manually close a request. Requests auto-expire after 72 hours.

## Verify API key

```
GET /api/v1/whoami
```

Check your API key is valid.

```json
{
  "keyId": "cmxxx...",
  "name": "My Agent Key",
  "type": "client"
}
```

## Rate limits

**Per IP (Platform edge middleware):**

| Route | Limit |
|-------|-------|
| `/api/v1/*` | 60 req/min per IP |
| Legacy polling (`GET /api/v1/help/*`) | 30 req/min per IP |
| Events polling (`GET /api/v1/events/*`) | 300 req/min per IP |

Each API key also has a per-key rate limit (default: 150 req/min, configurable in dashboard).

## Error format

```json
{ "error": "Request not found" }
```

| Code | Meaning |
|------|---------|
| `400` | Validation error |
| `401` | Missing or invalid API key |
| `403` | Key not authorized for this resource |
| `404` | Not found |
| `413` | Payload too large (>1MB) |
| `422` | Content blocked by safety filter (PII, XSS, etc.) |
| `429` | Rate limited |
