# Changelog

## v0.2.0 -- 2026-04-03

### Changed
- Renamed "Provider" to "Expert" throughout the platform -- all API endpoints, database schema, SDK methods, and documentation now use "expert" terminology
- API key prefix changed from `hs_prov_` to `hs_exp_`
- Dashboard route changed from `/dashboard/providers` to `/dashboard/experts`
- Consumer SDK: `ProviderStore` -> `ExpertStore`, `add-provider` -> `add-expert`, `list-providers` -> `list-experts`

---

## Approval Buttons -- 2026-04-04

### Added
- **Approval buttons in Telegram**: Requests with `requiresApproval: true` send inline Approve/Deny buttons via Telegram `InlineKeyboardMarkup`. Callback queries update the request and confirm the decision in-chat.
- **Approval buttons in Slack**: Requests with `requiresApproval: true` send Block Kit action buttons (green Approve, red Deny). Clicking a button updates the original message to show the decision and removes the buttons.
- **OpenClaw approval webhook**: New `POST /api/adapters/openclaw/:channelId/webhook` endpoint receives `approve`, `deny`, and `reply` actions from OpenClaw. Supports HMAC-SHA256 signature verification via `X-OpenClaw-Signature` header.
- **Consumer SDK `requiresApproval` option**: `submitRequest({ requiresApproval: true })` triggers approval buttons instead of free-text responses. Decisions are delivered as message events, polled via `getPendingEvents()`.
- **Database fields**: `requiresApproval` (boolean) and `approvalDecision` (string, `"approved"` or `"denied"`) on the HelpRequest model.

## Dashboard Summoning Context -- 2026-04-03

### Added
- **Summoning context in the dashboard client wizard**: The "Create New Client" wizard (Step 2) now includes summoning context presets (Strict, Budget-conscious, Safety-first), recently used contexts, and a custom textarea (500-char limit) -- matching the onboarding wizard.
- **Improved client-created dialog (Step 3)**: Shows full setup instructions with intro text, summoning guidelines (when set), and the setup link. The "Copy" button now copies the complete instructional text block instead of just the URL.
- Shared `SUMMON_CONTEXT_PRESETS` constant extracted to `src/lib/summon-context-presets.ts` for DRY use across both wizards.

## Per-Client Summoning Context -- 2026-03-30

### Changed
- **Summoning context is now per-client**: Each client connection can have its own summoning guidelines (up to 500 characters), set during the onboarding wizard. Previously, context was set on the expert profile.
- Summoning context is passed directly in `POST /api/v1/setup-link` body and stored on the `SetupToken`.
- Recently used contexts are saved to `recentSummonContexts` on the expert profile (JSON array, max 10, most recent first) for quick reuse.
- Removed summoning context editor from the dashboard expert detail panel.
- `GET /api/v1/experts` no longer includes `summonContext` in the response.
- `PATCH /api/experts/:id` no longer accepts `summonContext`.
- Presets (Strict, Budget-conscious, Safety-first) and recently used contexts appear in the wizard.

## Docker Refactor â€” 2026-03-27

### Changed
- **Breaking:** Merged Guard content safety into the main app as built-in middleware. The separate `heysummon-guard` Docker image is no longer needed.
- Simplified Docker deployment: just app + PostgreSQL (no more Guard proxy, keygen service, or shared key volumes)
- Content safety (HTML sanitization, URL defanging, PII detection/blocking) now runs in-process on API routes
- Removed Ed25519 receipt signing system (`x-guard-receipt` headers no longer used)
- Single Docker image: `ghcr.io/thomasansems/heysummon`

### Removed
- `ghcr.io/thomasansems/heysummon-guard` Docker image (deprecated)
- `Dockerfile.allinone` (superseded by single Dockerfile)
- `docker-compose.prod.yml`, `docker-compose.cloudflare.yml`, `docker-compose.ngrok.yml`, `docker-compose.tailscale.yml` (superseded by profiles in `docker-compose.yml`)
- Environment variables: `GUARD_SIGNING_KEY`, `GUARD_PUBLIC_KEY`, `REQUIRE_GUARD`, `GUARD_PORT`
- `guard/` directory and `scripts/generate-guard-keys.ts`

### Migration
If upgrading from a previous version:
1. Pull the new `docker-compose.yml`
2. Remove `GUARD_*` and `REQUIRE_GUARD` from your `.env`
3. Run `docker compose up -d` â€” the guard-keygen and heysummon-guard services are no longer defined
