# API Reference

Base URL: `http://your-instance:3445` (Docker) or `http://localhost:3435` (NPX/CLI)

All endpoints require `x-api-key` authentication unless noted.

---

## Help Requests

### Submit a request
`POST /api/v1/help`

| Field | Type | Required | Description |
|-------|------|----------|-------------|
| `apiKey` | string | ✅ | Client key (`hs_live_*`) |
| `question` | string | ✅ | Question for the human expert |
| `signPublicKey` | string | ❌ | Ed25519 public key (enables E2E) |
| `encryptPublicKey` | string | ❌ | X25519 public key (enables E2E) |
| `providerName` | string | ❌ | Route to a specific provider |
| `requiresApproval` | boolean | ❌ | Provider must approve before responding |
| `messages` | string | ❌ | Encrypted conversation history |
| `messageCount` | number | ❌ | Count of messages in history |

**Response `200`:**
```json
{ "requestId": "cm...", "refCode": "HS-A1B2C3D4", "status": "pending" }
```

---

### Get request status
`GET /api/v1/help/:requestId`

**Header:** `x-api-key: hs_live_*`

**Response `200`:**
```json
{
  "requestId": "cm...",
  "refCode": "HS-A1B2C3D4",
  "status": "responded",
  "response": "No, do not delete it.",
  "respondedAt": "2026-02-28T14:00:00Z"
}
```

---

### Get request by ref code
`GET /api/v1/requests/by-ref/:refCode`

**Header:** `x-api-key`

Look up any request using its `HS-XXXXXXXX` reference code.

## Key Exchange

### Exchange public keys
`POST /api/v1/key-exchange/:requestId`

**Header:** `x-api-key: hs_prov_*`

Provider sends their public keys to enable E2E encryption with the consumer.

```json
{
  "signPublicKey": "hex-encoded-Ed25519-public-key",
  "encryptPublicKey": "hex-encoded-X25519-public-key"
}
```

**Response `200`:**
```json
{ "success": true }
```

## Messages

**Header:** `x-api-key`


Or encrypted:

```json
{
  "from": "consumer",
  "ciphertext": "...",
  "iv": "...",
  "authTag": "...",
  "signature": "...",
  "messageId": "uuid"
}
```

`GET /api/v1/messages/:requestId`

**Header:** `x-api-key`

## Conversations

### Close a request
`POST /api/v1/close/:requestId`

**Header:** `x-api-key: hs_prov_*`

Closes the request. Provider key required.

---

### Rate a response
`POST /api/v1/rate/:requestId`

**Header:** `x-api-key`

```json
{ "rating": 4, "feedback": "Very helpful, thank you" }
```

**Response `200`:**
```json
{ "success": true, "rating": 4 }
```

Rating is `1`–`5`. Each request can only be rated once — a second call returns `409`.

---

### Report timeout
`POST /api/v1/help/:requestId/timeout`

**Header:** `x-api-key`

Tells the server the consumer gave up waiting. The request stays visible on the
dashboard so the provider can still respond later.

## Provider availability

Providers can configure availability windows. If a request is submitted outside
the provider's available hours, the API returns:

```json
{
  "rejected": true,
  "nextAvailableAt": "2026-03-27T09:00:00Z",
  "message": "Provider is currently unavailable. Next available at 09:00 CET."
}
```

Availability is configured per provider profile in the dashboard:
- **Quiet hours** — start/end time in the provider's timezone
- **Available days** — which days of the week the provider accepts requests

## Summoning context

Summoning context is **per-client** -- each client connection can have its own advisory
text (max 500 characters) that tells AI agents when to summon the provider. The context
is set during client setup and stored on the `SetupToken`:

1. Provider enters context in the onboarding wizard when connecting a client
2. Passed in the `POST /api/v1/setup-link` body as `summonContext`
3. Stored on `SetupToken.summonContext` for that specific client
4. Embedded as `HEYSUMMON_SUMMON_CONTEXT` in the consumer's `.env`
5. Referenced by the skill's "When to summon" section at runtime

Recently used contexts are saved to the provider's `recentSummonContexts` (JSON array,
max 10 entries, most recent first) for quick reuse when setting up new clients.

| Field | Location | Type | Max | Description |
|-------|----------|------|-----|-------------|
| `summonContext` | `POST /api/v1/setup-link` body | string \| null | 500 | Per-client advisory guidelines for AI |
| `recentSummonContexts` | `GET /api/providers/:id` response | string (JSON) \| null | 10 entries | Recently used contexts for quick reuse |

## Approval Buttons

When `requiresApproval: true` is set on a help request, the provider receives Approve/Deny buttons instead of a free-text reply prompt. Buttons are delivered via Telegram inline keyboard, Slack Block Kit actions, or OpenClaw webhook actions, depending on the provider's active channel.

After the provider clicks a button, the request transitions to `responded` with `approvalDecision` set to `"approved"` or `"denied"`. The decision is also stored as a Message record so consumers can detect it via `GET /api/v1/events/pending`.

Each request can only receive one decision. Double-clicks and subsequent presses return the existing decision without changing state.

### OpenClaw callback webhook

`POST /api/adapters/openclaw/:channelId/webhook`

Receives approval decisions (and replies) from OpenClaw. The channel ID is provided during OpenClaw channel activation.

**Headers:**

| Header | Required | Description |
|--------|----------|-------------|
| `X-OpenClaw-Signature` | Yes | HMAC-SHA256 hex digest of the request body, using the channel's webhook secret |
| `Content-Type` | Yes | `application/json` |

**Body:**

```json
{
  "action": "approve",
  "requestId": "cm...",
  "message": "optional reply text (for action=reply only)"
}
```

| Field | Type | Required | Description |
|-------|------|----------|-------------|
| `action` | string | Yes | `"approve"`, `"deny"`, or `"reply"` |
| `requestId` | string | Yes | The help request ID |
| `message` | string | No | Reply text (max 10,000 chars, required when `action` is `"reply"`) |

**Signature verification:**

```
HMAC-SHA256(webhookSecret, rawRequestBody) === X-OpenClaw-Signature header
```

The webhook secret is generated when the OpenClaw channel is activated and displayed in the channel setup instructions.

**Response `200` (approve/deny):**

```json
{
  "ok": true,
  "message": "Request HS-A1B2C3D4 -- Decision: Approved",
  "decision": "approved"
}
```

**Response `200` (reply):**

```json
{
  "ok": true,
  "message": "Reply sent for HS-A1B2C3D4"
}
```

**Error responses:**

| Code | Meaning |
|------|---------|
| `400` | Invalid payload or missing required fields |
| `403` | Invalid or missing signature |
| `404` | Channel or request not found |

## Webhooks

HeySummon can send webhook notifications to external endpoints when events occur.

**Payload:**

```json
{
  "type": "new_request",
  "requestId": "cm...",
  "refCode": "HS-A1B2C3D4",
  "question": "Should I proceed?"
}
```

**Event types:** `new_request`, `new_message`, `responded`, `closed`, `client_timeout`, `keys_exchanged`

**Security:** Every webhook includes an `X-HeySummon-Signature` header containing
an HMAC-SHA256 signature of the payload body. Verify this against your webhook secret
before trusting the payload.

## Request lifecycle

Requests move through these states:

```
pending → reviewing → active → responded → closed
                                    ↗
pending → expired (72h TTL)
pending → cancelled
```

| State | Meaning |
|-------|---------|
| `pending` | Submitted, waiting for provider |
| `reviewing` | Provider has opened the request |
| `active` | Provider is composing a response |
| `responded` | Provider sent a response |
| `closed` | Consumer acknowledged and closed |
| `expired` | No response within 72 hours |
| `cancelled` | Explicitly cancelled |
