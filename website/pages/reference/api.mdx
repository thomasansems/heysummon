# API Reference

Base URL: `http://your-instance:3445` (Docker) or `http://localhost:3435` (NPX/CLI)

All endpoints require `x-api-key` authentication unless noted.

## Key Exchange

### Exchange public keys
`POST /api/v1/key-exchange/:requestId`

**Header:** `x-api-key: hs_prov_*`

Provider sends their public keys to enable E2E encryption with the consumer.

```json
{
  "signPublicKey": "hex-encoded-Ed25519-public-key",
  "encryptPublicKey": "hex-encoded-X25519-public-key"
}
```

**Response `200`:**
```json
{ "success": true }
```

## Messages

### Send a message
`POST /api/v1/message/:requestId`

**Header:** `x-api-key`

```json
{
  "from": "consumer",
  "plaintext": "Follow-up question here"
}
```

Or encrypted:

```json
{
  "from": "consumer",
  "ciphertext": "...",
  "iv": "...",
  "authTag": "...",
  "signature": "...",
  "messageId": "uuid"
}
```

### List messages
`GET /api/v1/messages/:requestId`

**Header:** `x-api-key`

Returns all messages in the conversation, ordered by creation time.

## Provider availability

Providers can configure availability windows. If a request is submitted outside
the provider's available hours, the API returns:

```json
{
  "rejected": true,
  "nextAvailableAt": "2026-03-27T09:00:00Z",
  "message": "Provider is currently unavailable. Next available at 09:00 CET."
}
```

Availability is configured per provider profile in the dashboard:
- **Quiet hours** â€” start/end time in the provider's timezone
- **Available days** â€” which days of the week the provider accepts requests

## Summoning context

Providers can set optional advisory text (`summonContext`) that tells AI agents when
to summon them. This is stored on the provider profile (max 500 characters) and flows
through the system:

1. Provider writes context via the dashboard or wizard
2. Saved to `UserProfile.summonContext` via `PATCH /api/providers/:id`
3. Copied to `SetupToken.summonContext` when a setup link is created
4. Embedded as `HEYSUMMON_SUMMON_CONTEXT` in the consumer's `.env`
5. Referenced by the skill's "When to summon" section at runtime

| Field | Type | Max | Description |
|-------|------|-----|-------------|
| `summonContext` | string \| null | 500 | Advisory guidelines for AI on when to summon this provider |

The field is available on:
- `PATCH /api/providers/:id` (set/update)
- `GET /api/providers/:id` (read)
- `GET /api/v1/providers` (consumer-facing, read)

## Webhooks

HeySummon can send webhook notifications to external endpoints when events occur.

**Payload:**

```json
{
  "type": "new_request",
  "requestId": "cm...",
  "refCode": "HS-A1B2C3D4",
  "question": "Should I proceed?"
}
```

**Event types:** `new_request`, `new_message`, `responded`, `closed`, `client_timeout`, `keys_exchanged`

**Security:** Every webhook includes an `X-HeySummon-Signature` header containing
an HMAC-SHA256 signature of the payload body. Verify this against your webhook secret
before trusting the payload.

## Request lifecycle

Requests move through these states:

```
pending â†’ reviewing â†’ active â†’ responded â†’ closed
                                    â†—
pending â†’ expired (72h TTL)
pending â†’ cancelled
```

| State | Meaning |
|-------|---------|
| `pending` | Submitted, waiting for provider |
| `reviewing` | Provider has opened the request |
| `active` | Provider is composing a response |
| `responded` | Provider sent a response |
| `closed` | Consumer acknowledged and closed |
| `expired` | No response within 72 hours |
| `cancelled` | Explicitly cancelled |
