# API Reference

Base URL: `http://your-instance:3445` (Docker) or `http://localhost:3435` (NPX/CLI)

All endpoints require `x-api-key` authentication unless noted.

<Callout type="info">
**Privacy-first architecture.** E2E encryption is always on by default. The consumer SDK auto-generates Ed25519 + X25519 keypairs on every request. Key exchange with the expert is automatic. Your conversations are encrypted end-to-end ŌĆö even HeySummon cannot read them.

The initial question is submitted in plaintext so the content safety layer (Guard) can screen for PII, injection attacks, and abuse. After key exchange, all conversation messages are encrypted.

To disable E2E for debugging, pass `e2e: false` when creating the SDK client.
</Callout>

---

## Help Requests

HeySummon supports two modes on `/api/v1/help`:

- **Help mode** (default, `responseRequired: true`) ŌĆö expert replies; consumer polls or blocks for the answer.
- **Notification mode** (`responseRequired: false`) ŌĆö expert is alerted but no reply is expected. Terminates when the expert clicks **Acknowledge** in the dashboard, Telegram, Slack, or OpenClaw. 7-day TTL. Consumer SDK: use `notify()` as a sugar over `help()` ŌĆö see [Consumer API Reference](/consumer/api-reference#notification-mode-notify).

### Submit a request
`POST /api/v1/help`

| Field | Type | Required | Description |
|-------|------|----------|-------------|
| `apiKey` | string | Ō£ģ | Client key (`hs_cli_*`) |
| `question` | string | Ō£ģ | Question for the human expert (plaintext ŌĆö validated by Guard) |
| `signPublicKey` | string | | Ed25519 public key (hex DER). Auto-generated by SDK if not provided. |
| `encryptPublicKey` | string | | X25519 public key (hex DER). Auto-generated by SDK if not provided. |
| `expertName` | string | | Route to a specific expert |
| `requiresApproval` | boolean | | Expert must approve before responding |
| `responseRequired` | boolean | | When `false`, creates a notification-mode request ŌĆö expert is alerted, no reply expected. Defaults to `true`. |
| `messages` | string | | Encrypted conversation history |
| `messageCount` | number | | Count of messages in history |

When using the consumer SDK with `e2e: true` (the default), keys are generated automatically and stored in memory. You only need to provide `signPublicKey` and `encryptPublicKey` if you manage your own keys.

**Response `200` (help request):**
```json
{ "requestId": "cm...", "refCode": "HS-A1B2C3D4", "status": "pending" }
```

**Response `200` (notification ŌĆö `responseRequired: false`):**
```json
{
  "requestId": "cm...",
  "refCode": "HS-A1B2C3D4",
  "status": "pending",
  "responseRequired": false,
  "expiresAt": "2026-04-30T10:00:00.000Z"
}
```

Notification-mode requests skip phone-first / Twilio escalation and use a longer
7-day TTL. `serverPublicKey` is omitted (no encrypted reply is coming).

## Key Exchange

### Exchange public keys
`POST /api/v1/key-exchange/:requestId`

**Header:** `x-api-key: hs_exp_*`

Completes the E2E key exchange. When using the expert dashboard, this happens automatically when the expert opens a request. The dashboard generates its own Ed25519 + X25519 keypairs via the Web Crypto API and posts them here.

After this call, both sides have the other's public keys. Each side computes a shared secret via X25519 Diffie-Hellman, then derives per-message AES-256-GCM keys using HKDF-SHA256. All subsequent messages are encrypted end-to-end.

```json
{
  "signPublicKey": "hex-encoded-Ed25519-public-key",
  "encryptPublicKey": "hex-encoded-X25519-public-key"
}
```

**Response `200`:**
```json
{ "success": true }
```

## Messages

### Send a message
`POST /api/v1/message/:requestId`

**Header:** `x-api-key`

Not allowed on notification-mode requests (`responseRequired: false`) ŌĆö returns
`409` with code `NO_RESPONSE_REQUIRED`. Use `/acknowledge` to finalize a
notification instead.

When E2E is active (the default), the consumer SDK encrypts messages automatically using the expert's public key. The wire format is:

```json
{
  "from": "consumer",
  "ciphertext": "base64-AES-256-GCM-ciphertext",
  "iv": "base64-12-byte-IV",
  "authTag": "base64-16-byte-GCM-tag",
  "signature": "base64-Ed25519-signature-over-ciphertext",
  "messageId": "uuid"
}
```

When E2E is disabled (`e2e: false`) or before key exchange completes, messages are sent as plaintext:

```json
{
  "from": "consumer",
  "plaintext": "Follow-up question here"
}
```

---

### List messages
`GET /api/v1/messages/:requestId`

**Header:** `x-api-key`

Returns all messages in the conversation plus the public keys for both parties. The consumer SDK uses these keys to auto-decrypt messages.

**Response `200`:**
```json
{
  "messages": [
    {
      "id": "cm...",
      "from": "expert",
      "ciphertext": "...",
      "iv": "...",
      "authTag": "...",
      "signature": "...",
      "messageId": "uuid",
      "createdAt": "2026-03-01T10:00:00Z"
    }
  ],
  "consumerSignPubKey": "hex-Ed25519-public-key",
  "consumerEncryptPubKey": "hex-X25519-public-key",
  "expertSignPubKey": "hex-Ed25519-public-key",
  "expertEncryptPubKey": "hex-X25519-public-key"
}
```

Messages with `iv: "plaintext"` are unencrypted (e.g. Telegram replies or pre-key-exchange messages) and should be displayed as-is.

## Conversations

### Close a request
`POST /api/v1/close/:requestId`

**Header:** `x-api-key: hs_exp_*`

Closes the request. Expert key required.

For notification-mode requests (`responseRequired: false`), close returns `409`
with code `NOT_APPLICABLE` ŌĆö use `/acknowledge` instead. Calls against
already-`acknowledged` or `expired` notifications remain idempotent (`200`).

---

### Acknowledge a notification
`POST /api/v1/acknowledge/:requestId`

**Auth:** session cookie (dashboard) OR `x-api-key` (expert key or consumer key
scoped to the request).

Marks a notification-mode request (`responseRequired: false`) as seen. Idempotent
ŌĆö re-acking returns the original timestamp.

**Response `200`:**
```json
{ "success": true, "status": "acknowledged", "acknowledgedAt": "2026-04-23T10:00:00.000Z" }
```

**Errors:**
- `404` ŌĆö request not found or not owned by the caller
- `409` ŌĆö code `NOT_APPLICABLE` when the request expects a reply (use `/close`)
- `401` ŌĆö missing/invalid auth

---

### Rate a response
`POST /api/v1/rate/:requestId`

**Header:** `x-api-key`

```json
{ "rating": 4, "feedback": "Very helpful, thank you" }
```

**Response `200`:**
```json
{ "success": true, "rating": 4 }
```

Rating is `1`ŌĆō`5`. Each request can only be rated once ŌĆö a second call returns `409`.

---

### Report timeout
`POST /api/v1/help/:requestId/timeout`

**Header:** `x-api-key`

Tells the server the consumer gave up waiting. The request stays visible on the
dashboard so the expert can still respond later.

## Events

### Pending events
`GET /api/v1/events/pending`

**Header:** `x-api-key`

Returns undelivered events (up to 50). Use after reconnecting to catch up.

**Event types:**

| Type | Delivered to | Meaning |
|------|-------------|---------|
| `new_request` | expert | New help request waiting |
| `new_message` | expert + consumer | New message in an existing conversation |
| `keys_exchanged` | consumer | Expert completed key exchange |
| `responded` | consumer | Expert sent a response |
| `closed` | consumer | Request was closed |
| `cancelled` | consumer | Request was cancelled |
| `new_notification` | expert | New notification-mode request (`responseRequired: false`) |
| `notification_acknowledged` | consumer | Expert acknowledged a notification; includes `acknowledgedAt` |
| `notification_expired` | consumer | Notification passed its 7-day TTL without ack; includes `expiredAt` |

## Setup

### Verify install round-trip
`POST /api/v1/setup/verify-roundtrip`

**Header:** `x-api-key: hs_cli_*`

Confirms that a freshly-installed client can resolve a real notification channel
for its expert, end-to-end, **without sending an actual notification**. The route
creates a synthetic `probe: true` HelpRequest, runs the same expert + channel
resolution path the live `/api/v1/help` endpoint uses, and immediately closes the
row. It never dispatches Telegram, Slack, or OpenClaw fan-out ŌĆö it only proves
the channel *could* dispatch.

<Callout type="info">
Probe rows are filtered out of every dashboard, events, search, and channel
adapter query via the `nonProbe()` scope helper. They will not appear in the
expert's inbox, in `events/pending`, in metrics, or in GDPR data exports.
</Callout>

**Request body:** none. The request is fully scoped by the `x-api-key` header.

**Response `200`:**
```json
{ "ok": true, "expertName": "Jane Smith", "channelType": "telegram" }
```

**Errors:**

| Code | HTTP | Meaning |
|------|------|---------|
| `auth` | 401 | API key is invalid, expired, or inactive. |
| `expert_disabled` | 404 | Expert profile is missing or currently disabled. |
| `no_channel` | 422 | Expert has no active notification channel ŌĆö add one in the dashboard before installing. |
| `encryption_unconfigured` | 422 | Expert has channels but none are fully configured (missing tokens or transport secrets, e.g. Telegram `botToken`/`expertChatId`, Slack `botToken`/`channelId`, OpenClaw `webhookUrl`/`apiKey`). |
| `internal` | 500 | Unexpected server error while verifying. |

Error response shape:
```json
{ "ok": false, "code": "no_channel", "error": "Expert has no active notification channel..." }
```

This endpoint coexists with the older dashboard-polling `setup/verify` endpoint;
the install script should prefer `verify-roundtrip` because it exercises the real
channel-resolution path.

## Expert availability

Experts can configure availability windows. If a request is submitted outside
the expert's available hours, the API returns:

```json
{
  "rejected": true,
  "nextAvailableAt": "2026-03-27T09:00:00Z",
  "message": "Expert is currently unavailable. Next available at 09:00 CET."
}
```

Availability is configured per expert profile in the dashboard:
- **Quiet hours** ŌĆö start/end time in the expert's timezone
- **Available days** ŌĆö which days of the week the expert accepts requests

## Summoning context

Summoning context is **per-client** -- each client connection can have its own advisory
text (max 500 characters) that tells AI agents when to summon the expert. The context
is set during client setup and stored on the `SetupToken`:

1. Expert enters context in the onboarding wizard when connecting a client
2. Passed in the `POST /api/v1/setup-link` body as `summonContext`
3. Stored on `SetupToken.summonContext` for that specific client
4. Embedded as `HEYSUMMON_SUMMON_CONTEXT` in the consumer's `.env`
5. Referenced by the skill's "When to summon" section at runtime

Recently used contexts are saved to the expert's `recentSummonContexts` (JSON array,
max 10 entries, most recent first) for quick reuse when setting up new clients.

| Field | Location | Type | Max | Description |
|-------|----------|------|-----|-------------|
| `summonContext` | `POST /api/v1/setup-link` body | string \| null | 500 | Per-client advisory guidelines for AI |
| `recentSummonContexts` | `GET /api/experts/:id` response | string (JSON) \| null | 10 entries | Recently used contexts for quick reuse |

## Approval Buttons

When `requiresApproval: true` is set on a help request, the expert receives Approve/Deny buttons instead of a free-text reply prompt. Buttons are delivered via Telegram inline keyboard, Slack Block Kit actions, or OpenClaw webhook actions, depending on the expert's active channel.

After the expert clicks a button, the request transitions to `responded` with `approvalDecision` set to `"approved"` or `"denied"`. The decision is also stored as a Message record so consumers can detect it via `GET /api/v1/events/pending`.

Each request can only receive one decision. Double-clicks and subsequent presses return the existing decision without changing state.

---

### OpenClaw callback webhook

`POST /api/adapters/openclaw/:channelId/webhook`

Receives approval decisions (and replies) from OpenClaw. The channel ID is provided during OpenClaw channel activation.

**Headers:**

| Header | Required | Description |
|--------|----------|-------------|
| `X-OpenClaw-Signature` | Yes | HMAC-SHA256 hex digest of the request body, using the channel's webhook secret |
| `Content-Type` | Yes | `application/json` |

**Body:**

```json
{
  "action": "approve",
  "requestId": "cm...",
  "message": "optional reply text (for action=reply only)"
}
```

| Field | Type | Required | Description |
|-------|------|----------|-------------|
| `action` | string | Yes | `"approve"`, `"deny"`, or `"reply"` |
| `requestId` | string | Yes | The help request ID |
| `message` | string | No | Reply text (max 10,000 chars, required when `action` is `"reply"`) |

**Signature verification:**

```
HMAC-SHA256(webhookSecret, rawRequestBody) === X-OpenClaw-Signature header
```

The webhook secret is generated when the OpenClaw channel is activated and displayed in the channel setup instructions.

**Response `200` (approve/deny):**

```json
{
  "ok": true,
  "message": "Request HS-A1B2C3D4 -- Decision: Approved",
  "decision": "approved"
}
```

**Response `200` (reply):**

```json
{
  "ok": true,
  "message": "Reply sent for HS-A1B2C3D4"
}
```

**Error responses:**

| Code | Meaning |
|------|---------|
| `400` | Invalid payload or missing required fields |
| `403` | Invalid or missing signature |
| `404` | Channel or request not found |

## Webhooks

HeySummon can send webhook notifications to external endpoints when events occur.

**Payload:**

```json
{
  "type": "new_request",
  "requestId": "cm...",
  "refCode": "HS-A1B2C3D4",
  "question": "Should I proceed?"
}
```

**Event types:** `new_request`, `new_message`, `responded`, `closed`, `client_timeout`, `keys_exchanged`, `new_notification`, `notification_acknowledged`, `notification_expired`

**Security:** Every webhook includes an `X-HeySummon-Signature` header containing
an HMAC-SHA256 signature of the payload body. Verify this against your webhook secret
before trusting the payload.

## Request lifecycle

Help requests (`responseRequired: true`, the default) move through these states:

```
pending ŌåÆ reviewing ŌåÆ active ŌåÆ responded ŌåÆ closed
                                    ŌåŚ
pending ŌåÆ expired (72h TTL)
pending ŌåÆ cancelled
```

Notification-mode requests (`responseRequired: false`) use a simpler lifecycle:

```
pending ŌåÆ acknowledged
pending ŌåÆ expired (7d TTL)
```

| State | Meaning |
|-------|---------|
| `pending` | Submitted, waiting for expert |
| `reviewing` | Expert has opened the request |
| `active` | Expert is composing a response |
| `responded` | Expert sent a response |
| `closed` | Consumer acknowledged and closed |
| `acknowledged` | Expert acknowledged a notification (terminal, notification-mode only) |
| `expired` | Help requests: no response within 72 hours. Notifications: no ack within 7 days. |
| `cancelled` | Explicitly cancelled |
