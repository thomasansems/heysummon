# Changelog

## v0.2.18 -- 2026-04-08

### Fixed
- **NPX installer:** generated `.env` had `DATABASE_URL="file:./prisma/heysummon.db"`, which Prisma resolves relative to `schema.prisma` and ended up creating the SQLite file at `~/.heysummon/app/prisma/prisma/heysummon.db`. Migrations wrote there, but the runtime looked at the correct location and silently 500'd. Fresh installs effectively had no working database, leaving users stuck on the login page instead of the first-time signup screen
- **NPX installer:** `ensureAppEnv` now always overwrites `~/.heysummon/app/.env` from the canonical `~/.heysummon/.env`, so config edits actually take effect (previously the copy was skipped if the destination existed)
- **Auth flags endpoint:** `GET /api/auth/flags` now catches database errors and returns `dbError: true` instead of throwing a 500. The login page renders a clear "Database not reachable" message so a misconfigured install no longer silently degrades into an inert login form

### Breaking
- **NPX installer:** existing broken installs from earlier 0.2.x are not auto-migrated. Beta users with a nested `~/.heysummon/app/prisma/prisma/heysummon.db` should either reinstall (`rm -rf ~/.heysummon && npx @heysummon/app`) or manually move the file

## Unreleased

### Security
- **CRITICAL:** Activated Next.js security middleware -- rate limiting, CSP, CORS, and request size limits were defined in `src/proxy.ts` but never loaded by Next.js. Renamed to `src/middleware.ts` with correct export name so all protections are now active (resolves C1, C2, H1, H2, H3)
- **CRITICAL:** Removed hardcoded fallback database credentials from production `docker-compose.yml` -- now requires explicit `DB_USER`, `DB_PASSWORD`, `DB_NAME` via environment (resolves C3)
- Hardened CSP: removed `unsafe-eval` from `script-src` directive (resolves H4)
- Removed `user: root` from `db-migrate` Docker service to enforce least privilege (resolves H5)
- Moved E2E rate-limit bypass secrets from hardcoded CI workflow values to GitHub Actions secrets (resolves H6)
- Updated `vite` minimum to `>=6.4.2` in `landingspage` to fix arbitrary file read vulnerability (GHSA-p9ff-h696-f583, resolves H7)
- Replaced hardcoded `NEXTAUTH_SECRET` in E2E Docker Compose with required environment variable (resolves H8)

### Fixed
- NPX installer: fixed 404 error when downloading releases -- the GitHub API repo constant was set to `@heysummon/app` (npm package name) instead of `thomasansems/heysummon` (GitHub owner/repo format)
- CLI: reduced tagline animation from 6 cycles (~22s) to 2 cycles (~7s) so the init wizard starts faster
- Docker: all DB credentials (`DB_USER`, `DB_PASSWORD`, `DB_NAME`) are now parameterized via `.env` -- eliminates credential mismatch between `db` and `db-migrate` services on fresh deploys

### Changed
- `ask.sh` no longer auto-detects approval intent from keywords in the question. Approval mode now requires the explicit `--requires-approval` flag. This removes false positives where questions like "should I proceed?" silently became Approve/Deny prompts.
- Bash installers (`install.sh`, `install-local-dev.sh`) now display the branded HeySummon ASCII art banner, tagline, and typewriter animation matching the NPX installer experience

### Docs
- Replaced "Best Practice Prompts" page with "Summoning Guidelines" -- documents the wizard-based guidelines flow used during onboarding and client setup instead of static copy-paste prompts
- Updated "How It Works" page to document the client timeout (default 15 min) and fallback mechanism instead of only referencing the server-side 72h TTL
- Updated "Key Concepts" page with new Timeout and Fallback section explaining the three fallback behaviors (proceed cautiously, skip and continue, stop)
- Rewrote consumer API reference: fixed API key prefix (`hs_cli_*`), corrected whoami response format, added v3/v4 response distinction, documented 6 missing endpoints (by-ref lookup, timeout, rate, events/pending, events/ack, key-exchange), added curl/SDK CLI examples for all endpoints, added request lifecycle diagram, documented phone-first and expert unavailability behaviors
- Patched reference API docs for parity: fixed API key prefix, whoami response format, and v3/v4 status response distinction

## Approval Buttons -- 2026-04-04

### Added
- **Approval buttons in Telegram**: Requests with `requiresApproval: true` send inline Approve/Deny buttons via Telegram `InlineKeyboardMarkup`. Callback queries update the request and confirm the decision in-chat.
- **Approval buttons in Slack**: Requests with `requiresApproval: true` send Block Kit action buttons (green Approve, red Deny). Clicking a button updates the original message to show the decision and removes the buttons.
- **OpenClaw approval webhook**: New `POST /api/adapters/openclaw/:channelId/webhook` endpoint receives `approve`, `deny`, and `reply` actions from OpenClaw. Supports HMAC-SHA256 signature verification via `X-OpenClaw-Signature` header.
- **Consumer SDK `requiresApproval` option**: `submitRequest({ requiresApproval: true })` triggers approval buttons instead of free-text responses. Decisions are delivered as message events, polled via `getPendingEvents()`.
- **Database fields**: `requiresApproval` (boolean) and `approvalDecision` (string, `"approved"` or `"denied"`) on the HelpRequest model.

## Per-Client Summoning Context -- 2026-03-30

### Changed

## Docker Refactor â€” 2026-03-27

### Changed
- **Breaking:** Merged Guard content safety into the main app as built-in middleware. The separate `heysummon-guard` Docker image is no longer needed.

- `guard/` directory and `scripts/generate-guard-keys.ts`

1. Pull the new `docker-compose.yml`
2. Remove `GUARD_*` and `REQUIRE_GUARD` from your `.env`
3. Run `docker compose up -d` â€” the guard-keygen and heysummon-guard services are no longer defined
