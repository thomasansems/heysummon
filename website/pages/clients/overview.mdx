# Client Integrations

HeySummon connects AI coding agents to human experts. When an agent gets stuck, needs approval, or lacks context, it sends a help request to the expert and waits for a response.

HeySummon supports multiple AI platforms. Each platform has its own installation method, but the underlying API and scripts are the same.

## AI coding clients vs workflow orchestrators

This section covers **AI coding clients** -- tools an engineer drives at a keyboard, like [Claude Code](/clients/claude-code), [Codex CLI](/clients/codex), [Gemini CLI](/clients/gemini), and [OpenClaw](/clients/openclaw). The skill is installed once per machine and the agent calls HeySummon when it gets stuck mid-task.

If you are wiring HeySummon into a **workflow orchestrator** -- something that runs unattended jobs on a server, like n8n -- see [Integrations -> Orchestrators](/integrations/orchestrators/n8n) instead. The orchestrator nodes share the same Consumer API but install differently and run inside the orchestrator runtime.

*See also:* [Integrations](/integrations/orchestrators/n8n).

## How It Works

1. **Expert creates a setup link** from the HeySummon dashboard
2. **Client installs the skill** using the platform-specific method
3. **Agent uses the skill** when it needs human input -- it sends a question and blocks until the expert responds
4. **Expert responds** via Telegram, the dashboard, or their preferred channel

## What Gets Installed

Every platform gets the same core components:

- **SKILL.md** -- instructions the AI agent reads to understand how to use HeySummon
- **Scripts** -- bash wrappers around the `@heysummon/consumer-sdk` CLI:
  - `ask.sh` -- send a question and wait for the response (blocking)
  - `setup.sh` -- configure API key and base URL
  - `add-expert.sh` -- register an expert by API key
  - `list-experts.sh` -- list registered experts
  - `check-status.sh` -- check status of a request
- **.env** -- configuration file with API key, base URL, timeout, and summoning guidelines

## Two modes: summon vs notify

HeySummon supports two interaction patterns, both available from every client skill:

| Verb | Blocks the agent? | What the expert sees | Use when |
|------|------------------|---------------------|----------|
| `hey summon <name>` | Yes -- agent waits for a reply | Notification with a reply prompt (or Approve/Deny if `--requires-approval`) | The agent is stuck, needs approval, or needs judgment before continuing |
| `notify <name>` | No -- agent continues | Notification with a single "Acknowledge" button | Fire-and-forget: deploy finished, job ran, status heads-up |

Under the hood, `notify` calls the consumer SDK's `notify()` method (`responseRequired: false`). Notifications render with a single Acknowledge affordance across Telegram, Slack, OpenClaw, and the dashboard, and expire after 7 days if the expert never acks. See [Consumer API Reference -> Notification Mode](/consumer/api-reference#notification-mode-notify) for the full API surface.

## Summoning Guidelines

When creating a setup link, experts can define **summoning guidelines** -- rules that tell the AI when it should and shouldn't ask for help. These guidelines are stored in the `.env` file as `HEYSUMMON_SUMMON_CONTEXT` and the agent follows them automatically.

Examples:
- "Only summon when stuck and cannot proceed without human input"
- "Summon for decisions that could cost money or affect billing"
- "Summon before any destructive action like deleting data or dropping tables"
