# Content Safety

Content safety runs as built-in middleware inside the HeySummon app. Every incoming request is automatically scanned before processing.

---

## What it checks

Content safety validates `POST` requests to `/api/v1/help` and `/api/v1/message/*`.

| Check | What it does | Response on fail |
|-------|-------------|-----------------|
| **HTML/XSS sanitization** | Strips HTML tags, script injection, dangerous attributes | Cleaned |
| **PII detection** | Credit cards (Luhn-validated), SSNs (US), BSNs (Dutch) | `422` (blocked) |
| **Email detection** | Detects email address patterns | Redacted |
| **Phone detection** | International and local phone number formats | Redacted |
| **URL defanging** | Converts URLs to safe format (`hxxps://`, `[.]`) | Cleaned |

API key validation, IP binding, and rate limiting are separate middleware layers.

---

## OWASP ZAP tested

Every pull request runs an **OWASP ZAP API scan** against a live instance:

- Tests all `/api/v1/*` endpoints with SQLi, XSS, path traversal, and RCE payloads
- Fails CI if any HIGH severity vulnerability is found
- Report uploaded as artifact on every run

A separate **malicious content check** also runs in CI:

```bash
bash e2e/zap-malicious-check.sh
```

Tests verified:
- Credit card number -> blocked (422)
- SSN -> blocked (422)
- BSN (Dutch) -> blocked (422)
- XSS `<script>` -> sanitized
- SQL injection -> handled safely (no 500)
- No API key -> 401
- Message to nonexistent request -> 403/404

---

## Implementation

Content safety is implemented as two modules in the main app:

```
src/lib/
  content-safety.ts           # Content scanning (XSS, PII, URL defanging)
  content-safety-middleware.ts # Route-level middleware (extract text, check, block/pass)
```

The middleware extracts text from request bodies (`question`, `messages[].content`, `plaintext`), runs the safety pipeline, and either blocks the request (422) or passes sanitized content through to the route handler.
