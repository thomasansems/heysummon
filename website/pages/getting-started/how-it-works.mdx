# How It Works

HeySummon is a relay platform that connects AI agents to human experts in real time.

---

## Request lifecycle

```
pending  →  reviewing  →  responded
    │                  →  closed    (manually via API)
    │
    └── client timeout (default 15 min)
           → fallback behavior applies
           → request stays open for expert
           → server expires after TTL (default 72h)
```

| Status | Meaning |
|--------|---------|
| `pending` | Submitted, waiting for a human to open it |
| `reviewing` | Human has opened the request in the dashboard |
| `responded` | Human has replied |
| `expired` | Server-side TTL elapsed (default 72 hours) — background cleanup |
| `closed` | Manually closed via `/api/v1/close/:id` |

---

## Full flow

```
AI Agent
  │
  │  POST /api/v1/help
  │  { apiKey, question (encrypted), signPublicKey, encryptPublicKey }
  ▼
App
  │  Validates x-api-key
  │  Content safety: XSS, PII, malicious URLs
  │  Generates ref code: HS-XXXXXXXX
  │  Stores encrypted request
  │  Creates event on expert topic
  ▼
Expert dashboard / watcher (via polling)
                       │
                       │  Notification: [HS-XXXXXXXX] New request
                       │  Human reads, responds
                       ▼
                   Platform
                       │  Status → responded
                       │  Creates event on consumer topic
                       ▼
                   AI Agent (via polling)
                       │  GET /api/v1/help/:id → { status: "responded" }
                       │
                       │  OR: timeout reached (default 15 min)
                       │  POST /api/v1/help/:id/timeout
                       │  → receives fallback instruction
```

---

## Timeout and fallback

When a consumer submits a help request, the consumer SDK polls for a response up to a configurable **timeout** (default: **15 minutes**, configurable via the Setup Wizard). If no expert responds within that window, the SDK calls the timeout endpoint and receives a **fallback instruction** that tells the agent what to do next.

### Fallback behaviors

| Fallback | Behavior |
|----------|----------|
| **Proceed cautiously** (default) | Pick the safest available option. Document all decisions made without expert input and flag them for review. |
| **Skip and continue** | Skip the blocked task and continue with other work. Flag the skipped task for expert review later. |
| **Stop** | Halt all execution immediately. Do not proceed until an expert responds. |

These preferences are configured per client during setup and stored with the API key.

<Callout type="info">
  A client timeout does **not** close or expire the request. The request stays open on the server so an expert can still respond later. The agent simply moves on according to its fallback behavior. The expert is notified that the client timed out waiting.
</Callout>

### Server-side expiration

Independently of the client timeout, the server runs a background monitor that expires truly abandoned requests after a configurable TTL (default: **72 hours**, set via `HEYSUMMON_REQUEST_TTL_MS`). This is a cleanup mechanism — not the primary timeout that agents experience.

---

## Reference codes

Every request gets an `HS-XXXXXXXX` code (8 alphanumeric characters).

These codes are:
- Shown in the dashboard and notifications
- Used to reply from a messaging app: `[HS-A1B2C3D4] Yes, proceed`
- Included in audit logs for traceability

---

## Real-time vs polling

| Method | How | Latency |
|--------|-----|---------|
| **Event polling** | `GET /api/v1/events/pending` — poll for new events | Seconds |
| **Status polling** | `GET /api/v1/help/:id` — check every N seconds | N seconds |

Use event polling for production agents. Status polling is simpler for scripts and one-off requests.

---

## Multi-turn conversations

Requests support back-and-forth messages. After the initial response, the agent can send follow-up questions using `POST /api/v1/message/:requestId`. Each message gets a Mercure notification.

All messages in a conversation share the same `HS-XXXXXXXX` ref code.

---

## Content safety

All API requests are automatically scanned by built-in content safety middleware:

- Scans for malicious content: XSS, HTML injection, PII (credit cards, SSNs, BSNs)
- Defangs suspicious URLs
- Blocks requests containing credit cards or SSN/BSN patterns (422)

See [Security → Content Safety](/security/content-safety-guard) for details.
