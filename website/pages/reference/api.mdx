# API Reference

Base URL: `http://your-instance:3445` (Docker) or `http://localhost:3000` (NPX)

All endpoints require `x-api-key` authentication unless noted.

---

## Help Requests

### Submit a request
`POST /api/v1/help`

| Field | Type | Required | Description |
|-------|------|----------|-------------|
| `apiKey` | string | ✅ | Client key (`hs_live_*`) |
| `question` | string | ✅ | Question for the human expert |
| `signPublicKey` | string | ❌ | Ed25519 public key (enables E2E) |
| `encryptPublicKey` | string | ❌ | X25519 public key (enables E2E) |
| `providerName` | string | ❌ | Route to a specific provider |
| `requiresApproval` | boolean | ❌ | Provider must approve before responding |
| `messages` | string | ❌ | Encrypted conversation history |
| `messageCount` | number | ❌ | Count of messages in history |

**Response `200`:**
```json
{ "requestId": "cm...", "refCode": "HS-A1B2C3D4", "status": "pending" }
```

---

### Get request status
`GET /api/v1/help/:requestId`

**Header:** `x-api-key: hs_live_*`

**Response `200`:**
```json
{
  "requestId": "cm...",
  "refCode": "HS-A1B2C3D4",
  "status": "responded",
  "response": "No, do not delete it.",
  "respondedAt": "2026-02-28T14:00:00Z"
}
```

---

### Get request by ref code
`GET /api/v1/requests/by-ref/:refCode`

**Header:** `x-api-key`

Look up any request using its `HS-XXXXXXXX` reference code.

## Messages

### Send a message
`POST /api/v1/message/:requestId`

**Header:** `x-api-key`

```json
{
  "from": "consumer",
  "plaintext": "Follow-up question here"
}
```

Or encrypted:

```json
{
  "from": "consumer",
  "ciphertext": "...",
  "iv": "...",
  "authTag": "...",
  "signature": "..."
}
```

---

### List messages
`GET /api/v1/messages/:requestId`

**Header:** `x-api-key`

Returns all messages in the conversation, ordered by creation time.

---

## Conversations

### Close a request
`POST /api/v1/close/:requestId`

**Header:** `x-api-key: hs_prov_*`

Closes the request. Provider key required.

---

### Rate a response
`POST /api/v1/rate/:requestId`

**Header:** `x-api-key`

```json
{ "rating": 4, "feedback": "Very helpful, thank you" }
```

**Response `200`:**
```json
{ "success": true, "rating": 4 }
```

Rating is `1`–`5`. Each request can only be rated once — a second call returns `409`.

---

### Report timeout
`POST /api/v1/help/:requestId/timeout`

**Header:** `x-api-key`

Tells the server the consumer gave up waiting. The request stays visible on the
dashboard so the provider can still respond later.

---

## Events

### Pending events
`GET /api/v1/events/pending`

**Header:** `x-api-key`

Returns undelivered events (up to 50). Use after reconnecting to catch up.

---

### Acknowledge an event
`POST /api/v1/events/ack/:requestId`

**Header:** `x-api-key`

Mark an event as delivered.

## Provider availability

Providers can configure availability windows. If a request is submitted outside
the provider's available hours, the API returns:

```json
{
  "rejected": true,
  "nextAvailableAt": "2026-03-27T09:00:00Z",
  "message": "Provider is currently unavailable. Next available at 09:00 CET."
}
```

Availability is configured per provider profile in the dashboard:
- **Quiet hours** — start/end time in the provider's timezone
- **Available days** — which days of the week the provider accepts requests

## Webhooks

HeySummon can send webhook notifications to external endpoints when events occur.

**Payload:**

```json
{
  "type": "new_request",
  "requestId": "cm...",
  "refCode": "HS-A1B2C3D4",
  "question": "Should I proceed?"
}
```

**Event types:** `new_request`, `new_message`, `responded`, `closed`, `client_timeout`, `keys_exchanged`

**Security:** Every webhook includes an `X-HeySummon-Signature` header containing
an HMAC-SHA256 signature of the payload body. Verify this against your webhook secret
before trusting the payload.

## Request lifecycle

Requests move through these states:

```
pending → reviewing → active → responded → closed
                                    ↗
pending → expired (72h TTL)
pending → cancelled
```

| State | Meaning |
|-------|---------|
| `pending` | Submitted, waiting for provider |
| `reviewing` | Provider has opened the request |
| `active` | Provider is composing a response |
| `responded` | Provider sent a response |
| `closed` | Consumer acknowledged and closed |
| `expired` | No response within 72 hours |
| `cancelled` | Explicitly cancelled |

## Error codes

| Code | Meaning |
|------|---------|
| `400` | Validation error |
| `401` | Missing or invalid API key |
| `403` | Key not authorized for this resource |
| `404` | Not found |
| `413` | Payload too large (>1MB) |
| `422` | Content blocked by Guard (PII, XSS) |
| `429` | Rate limited |
| `500` | Internal server error |
